# 1. 性能优化指南

本文档介绍 Hermes Agent 的性能优化方法，帮助你在保证功能的前提下，提升响应速度、降低资源消耗、减少使用成本。

## 概述
Hermes Agent 已经内置了很多性能优化特性，大部分默认开启，但可以根据实际场景调整配置，获得更好的性能表现。优化方向主要包括：响应速度优化、资源占用优化、使用成本优化三个方面。

## 基础配置优化

### 1. 提示词缓存优化
提示词缓存是降低 Token 消耗、提升响应速度最有效的方式之一，默认已经开启，无需额外配置，支持的模型包括：
- Anthropic 全系列模型（Claude 3 Opus/Sonnet/Haiku）
- 所有兼容 Anthropic 缓存协议的代理服务

**优化配置**：
```yaml
model:
  cache_ttl: 3600 # 缓存有效期，单位秒，默认3600秒=1小时
  cache_enabled: true # 是否开启缓存，默认开启
```
对于长时间运行的固定任务场景，可以将 `cache_ttl` 调大到 86400（24小时），进一步提升缓存命中率。

**收益**：多轮对话场景下，输入 Token 消耗降低 60%~80%，响应速度提升 30%~50%。

### 2. 上下文压缩优化
上下文压缩可以在不丢失关键信息的前提下，大幅减少上下文长度，降低 Token 消耗，避免超出模型上下文窗口限制。

**优化配置**：
```yaml
session:
  auto_compress_context: true # 是否自动压缩，默认开启
  compression_threshold: 0.5 # 上下文长度达到模型窗口的50%时触发压缩，默认0.5
  compression_model: "anthropic/claude-3-haiku" # 用于压缩的低成本模型
```
**优化建议**：
- 对于长对话场景，可以将 `compression_threshold` 调低到 0.3，提前触发压缩，避免后续突发的 Token 暴涨
- 优先选择速度快、成本低的模型作为压缩模型，比如 Claude 3 Haiku、GPT-4o Mini 等
- 对于对信息完整性要求极高的场景，可以禁用自动压缩，手动通过 `/compress` 命令按需压缩

**收益**：长对话场景下，Token 消耗降低 50%~70%，支持的对话轮次提升 2~3 倍。

### 3. 模型配置优化
选择合适的模型可以在成本、速度、效果之间取得最佳平衡：

| 场景 | 推荐模型 | 优势 |
|------|----------|------|
| 日常聊天、简单工具调用 | Claude 3 Haiku / GPT-4o Mini | 速度快、成本低，足够应对大部分场景 |
| 复杂任务、代码开发、长文本处理 | Claude 3.5 Sonnet / GPT-4o | 平衡速度和效果，性价比最高 |
| 高难度任务、逻辑推理、专业领域 | Claude 3 Opus / GPT-4o Advanced | 效果最好，但速度慢、成本高 |

**优化配置**：
```yaml
model:
  default: "anthropic/claude-3-5-sonnet-20240620" # 主模型
  fallback_model: "anthropic/claude-3-haiku-20240307" # 降级模型，主模型不可用时自动切换
  max_tokens: 2048 # 根据实际场景调整，不需要太长响应可以调低，减少成本
  temperature: 0.7 # 不需要创意的场景可以调低到0，提升结果稳定性
```
**动态模型切换技巧**：
可以通过 `/model` 命令随时切换模型，比如：
- 常规对话使用 Haiku，成本低速度快
- 需要处理复杂任务时临时切换到 Sonnet
- 疑难问题再切换到 Opus

**收益**：合理选择模型可以降低 50%~90% 的使用成本，同时提升 20%~100% 的响应速度。

## 工具调用优化

### 1. 工具缓存配置
对于重复调用的工具，可以开启工具缓存，避免重复执行：
```yaml
tools:
  cache_enabled: true # 开启工具缓存，默认开启
  cache_ttl: 300 # 缓存有效期5分钟
  cache_max_size: 1000 # 最大缓存条目数
```
支持缓存的工具包括：
- 网络搜索、网页抓取工具（相同搜索词/URL 结果缓存）
- 文件读取工具（相同文件路径，内容未修改时缓存）
- 代码执行工具（相同代码和输入缓存）
- 所有只读类工具

### 2. 并行执行优化
Hermes Agent 自动支持安全工具的并行执行，无需额外配置，会自动判断：
- 多个无副作用的工具（文件读、搜索、网页抓取等）会并行执行
- 有副作用的工具（文件写、终端命令执行）串行执行
- 依赖其他工具结果的工具自动等待依赖完成

**优化技巧**：
在提问时可以一次提出多个独立的查询需求，Agent 会自动并行处理，大幅提升整体速度，比如：
```
同时帮我：
1. 读取当前目录下的 README.md
2. 搜索 Hermes Agent 最新的版本信息
3. 查看当前系统的 Python 版本
```
三个任务会并行执行，总耗时约等于单个最长任务的耗时，而不是三个任务的总和。

### 3. 工具权限配置
合理配置工具权限，减少不必要的安全检查开销：
```yaml
tools:
  exec_ask: false # 关闭命令执行审批，仅在开发环境和完全信任的场景下使用
  file_restrict: true # 保持文件路径限制，避免安全风险
  allowed_paths:
    - /home/user/workspace # 添加允许访问的路径，减少路径检查开销
```

## 部署层面优化

### 1. 资源配置
根据使用场景配置合适的资源：
| 场景 | 最低配置 | 推荐配置 |
|------|----------|----------|
| 个人使用，低并发 | 1核CPU，2GB内存 | 2核CPU，4GB内存 |
| 团队使用，10人以下 | 2核CPU，4GB内存 | 4核CPU，8GB内存 |
| 企业使用，10~100人 | 4核CPU，8GB内存 | 8核CPU，16GB内存 |
| 高并发生产环境 | 8核CPU，16GB内存 | 16核CPU，32GB内存，多实例部署 |

**内存优化配置**：
```yaml
session:
  max_history_length: 50 # 减少历史消息保留长度，降低内存占用
  auto_persist: true # 自动持久化会话到磁盘，减少内存缓存
persistence:
  wal_autocheckpoint: 1000 # SQLite WAL自动检查点频率，降低磁盘IO
```

### 2. 网络优化
- 部署在离模型 API 端点近的地域，降低网络延迟：
  - 访问 Anthropic API 推荐部署在 AWS 美西/弗吉尼亚区域
  - 访问 OpenAI API 推荐部署在 AWS 美东区域
- 配置代理加速，减少网络延迟，提升 API 调用成功率：
```env
HTTP_PROXY=http://proxy-server:port
HTTPS_PROXY=http://proxy-server:port
NO_PROXY=localhost,127.0.0.1,localaddress,.localdomain.com
```
- 增加 API 超时时间，避免网络波动导致的失败：
```yaml
model:
  timeout: 120 # 超时时间120秒，默认30秒
  max_retries: 5 # 最大重试次数，默认3次
  retry_delay: 2 # 重试间隔指数退避基础秒数
```

### 3. 生产环境部署优化
- 使用 Docker 容器部署，标准化运行环境，提升稳定性
- 配置反向代理和负载均衡，支持多实例部署，提升并发能力
- 开启 gzip 压缩，减少网络传输大小
- 使用 SSD 磁盘存储会话数据，提升 SQLite 读写性能
- 配置监控告警，及时发现性能瓶颈和异常

## 成本优化技巧

### 1. Token 消耗控制
- 调低 `max_tokens`，根据实际场景需求设置合理的最大响应长度，不需要长响应的场景可以设置为 1024 或更低
- 开启上下文压缩，减少历史消息的 Token 占用
- 及时使用 `/clear` 命令清空不需要的会话上下文，避免无效的 Token 消耗

### 2. 模型成本控制
- 优先使用低成本模型处理简单任务，高成本模型仅处理复杂任务
- 对于大量重复的任务，可以使用本地小模型预处理，减少大模型调用次数
- 利用模型价格差异：Claude 3 Haiku 成本仅为 Opus 的 1/30，效果可以满足 80% 以上的日常场景

### 3. 缓存利用
- 充分利用提示词缓存，相同的系统提示和上下文仅需要支付一次 Token 费用
- 开启工具缓存，避免重复执行相同的工具调用
- 对于常见问题，建立本地知识库，直接返回结果，不需要调用模型

## 性能监控与调优

### 内置性能统计
使用 `/status` 命令查看当前会话的性能统计：
```
会话ID: xxx
运行时长: 1小时23分
Token 消耗: 输入12345，输出6789，缓存命中8765
总调用次数: 模型调用23次，工具调用45次
平均响应时间: 2.3秒
```

### 日志分析
日志文件位于 `~/.hermes/logs/` 目录，包含详细的性能数据：
- 每个模型调用的耗时、Token 消耗、费用
- 每个工具执行的耗时、资源占用
- 缓存命中率、压缩率等优化指标

### 常见性能瓶颈排查
1. **响应速度慢**：
   - 检查网络延迟，是否能正常访问模型 API
   - 检查是否上下文过长，触发多次压缩
   - 尝试切换到速度更快的模型
2. **内存占用高**：
   - 检查是否有太多会话驻留内存，调低 `max_history_length`
   - 检查是否有后台任务内存泄漏，重启服务释放
3. **成本过高**：
   - 检查缓存命中率是否过低，调整缓存配置
   - 检查是否有大量不必要的长响应，调低 `max_tokens`
   - 切换到性价比更高的模型

## 最佳实践总结
1. **优先开启默认优化**：提示词缓存、自动压缩默认已经开启，不需要额外配置即可获得大部分优化收益
2. **合理选择模型**：根据任务复杂度动态选择合适的模型，平衡速度、成本、效果
3. **控制上下文长度**：及时清理不需要的上下文，避免无效的 Token 消耗
4. **利用并行执行**：一次提交多个独立任务，充分利用并行执行特性提升整体效率
5. **定期监控性能**：关注 Token 消耗、响应时间、缓存命中率等指标，持续优化
